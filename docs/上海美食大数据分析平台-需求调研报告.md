## 一、项目愿景文档

### 1.1 项目背景与战略对齐

随着互联网餐饮行业的快速发展,上海作为国际化大都市,美食市场竞争激烈,消费者对餐饮信息的需求日益增长。为了更好地理解上海美食市场动态,对餐厅的口碑、价格、菜系分布、消费趋势进行深入分析,本项目旨在通过采集多源美食数据,构建上海美食大数据分析平台,以支持餐饮从业者决策制定、消费者选择和政府监管。

**政策与行业依据**:
- 《"十四五"数字经济发展规划》: 推动数字技术与实体经济深度融合
- 《上海市促进在线新经济发展行动方案》: 支持餐饮行业数字化转型
- 《食品安全法》: 加强食品安全监管和信息公开
- 《个人信息保护法》《数据安全法》: 规范数据采集和使用

**市场需求输入**:
- 餐饮监管部门访谈记录: 《上海市市场监管局餐饮监管需求调研-20260215》
- 餐饮从业者问卷: 《上海餐饮企业数字化需求调研-58份有效样本》
- 消费者舆情分析: 大众点评/小红书"上海美食"相关话题热度分析
- 12345热线工单: 餐饮信息不透明投诉占比12%

### 1.2 项目范围界定

**包含范围**:
- 上海市16个行政区的餐饮数据采集与分析
- 大众点评、美团、小红书等公开平台数据
- 政府公开的食品安全监管数据
- 餐厅基础信息、评价、价格、菜系等维度分析
- 数据可视化与报告生成

**不包含范围**:
- 不涉及餐饮企业内部经营数据(财务、供应链等)
- 不提供餐饮预订和交易功能
- 不进行个人征信评估
- 不采集非公开的用户隐私数据

### 3.4 业务用例规约示例

#### 业务用例规约: BUC-002 生成美食市场分析报告

**用例ID**: BUC-002
**业务主角**: 市场研究员
**业务目标**: 在4小时内获得包含多源数据的上海美食市场分析报告

**1. 前置条件 (Pre-conditions)**
- 平台已完成T+1数据更新(每日凌晨2点后)
- 研究员拥有"高级分析"权限
- 数据覆盖率≥90%

**2. 主成功场景 (Main Success Scenario)**
1. 研究员选择分析维度(区域、菜系、价格区间、时间范围)
2. 系统自动聚合大众点评、美团、小红书及政府公示数据
3. 系统执行数据分析:
   - 区域餐厅密度分析
   - 菜系分布统计
   - 价格趋势分析
   - 评论情感分析
   - 热门菜品挖掘
4. 系统生成PDF/Word格式报告,包含图表和数据洞察
5. 系统推送报告至研究员邮箱
6. 研究员确认报告无误,归档至知识库

**3. 替代流程 (Alternative Flows)**
- 3a. 数据缺失: 若某区域数据覆盖率<80%,系统标记"数据不足",建议扩大时间范围或区域范围
- 4a. 生成失败: 若分析任务超时,系统自动重试1次,仍失败则通知运维人员
- 5a. 自定义需求: 研究员可自定义分析维度和图表类型

**4. 后置条件 (Post-conditions)**
- 业务产物: 一份可发布的《上海美食市场分析报告》
- 状态变更: 该分析任务标记为"已完成"
- KPI达成: 报告生成耗时≤4小时

**5. 业务规则**
- 报告必须包含数据来源说明和采集时间
- 敏感信息(用户姓名、电话)必须脱敏
- 报告数据保留期为2年

## 四、数据源盘点与采集方案

### 4.1 可爬取的网站及内容

#### ✅ 已确认数据源:

**1. 大众点评 (dianping.com)**
- 爬取内容:
  - 餐厅基础信息: 店名、地址、电话、营业时间、人均消费
  - 评分数据: 口味评分、环境评分、服务评分、综合评分
  - 用户评论: 评论内容、评论时间、用户等级、点赞数
  - 菜品信息: 招牌菜、菜品图片、菜品价格
  - 标签信息: 菜系、适合场景(约会/聚餐/商务)、特色服务
  - 地理位置: 经纬度坐标、所属商圈、地铁站距离
- 采集方式: Scrapy爬虫 + Selenium(处理动态加载)
- 更新频率: 每日增量采集新评论,每周全量更新餐厅信息

**2. 美团 (meituan.com)**
- 爬取内容:
  - 餐厅基础信息: 店名、地址、电话、营业时间、人均消费
  - 销量数据: 月销量、团购销量、外卖销量
  - 优惠信息: 团购套餐、优惠券、满减活动
  - 用户评价: 评论内容、评分、晒图数量
  - 配送信息: 配送范围、配送费、预计送达时间
- 采集方式: Scrapy爬虫 + API逆向
- 更新频率: 每日增量采集

**3. 小红书 (xiaohongshu.com)**
- 爬取内容:
  - 探店笔记: 标题、正文、发布时间、点赞数、收藏数
  - 图片/视频: 餐厅环境照、菜品照
  - 用户标签: #上海美食 #探店 #网红餐厅等话题标签
  - 用户互动: 评论内容、用户画像
- 采集方式: API接口 + 移动端抓包
- 更新频率: 每日采集热门笔记

**4. 抖音/快手 (短视频平台)**
- 爬取内容:
  - 探店视频: 视频标题、描述、发布时间、播放量、点赞数
  - POI信息: 关联的餐厅位置信息
  - 用户评论: 评论内容、互动数据
- 采集方式: 移动端API + 视频OCR文字提取
- 更新频率: 每周采集热门视频

**5. 饿了么/美团外卖**
- 爬取内容:
  - 外卖餐厅信息: 店名、地址、起送价、配送费
  - 菜品信息: 菜品名称、价格、月销量、好评率
  - 优惠活动: 满减、折扣、会员价
- 采集方式: 移动端API逆向
- 更新频率: 每日采集

**6. 政府公开数据**
- 上海市市场监督管理局官网:
  - 食品经营许可证信息
  - 食品安全抽检结果
  - 行政处罚公示
- 上海市统计局:
  - 餐饮行业统计数据
  - 居民消费支出数据
- 采集方式: 定向爬虫 + 人工下载
- 更新频率: 每月更新

**7. 企业信用信息公示系统**
- 爬取内容:
  - 餐饮企业工商注册信息
  - 企业信用评级
  - 经营异常名录
- 采集方式: Scrapy爬虫
- 更新频率: 每月更新

### 4.2 数据合规性要求

⚠️ **数据合规性要求**:
- 爬虫遵守 robots.txt,采集频率≤1次/秒
- 个人敏感信息(用户姓名、电话、身份证)必须脱敏
- 用户评论数据仅用于统计分析,不用于商业营销
- 所有数据标注来源与采集时间,支持溯源
- 遵守《个人信息保护法》《数据安全法》相关规定

### 4.3 业务流程梳理

**数据驱动决策闭环流程**:

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  数据采集   │─────>│  数据清洗   │─────>│  数据存储   │
│  (爬虫集群) │      │  (ETL处理)  │      │(MySQL/Mongo)│
└─────────────┘      └─────────────┘      └─────────────┘
                                                  │
                                                  ▼
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  决策执行   │<─────│  报告生成   │<─────│  数据分析   │
│(监管/经营)  │      │(可视化展示) │      │(统计/挖掘)  │
└─────────────┘      └─────────────┘      └─────────────┘
```

**关键业务规则**:
- 数据更新频率: 爬虫每日凌晨2点增量采集,T+1批处理
- 异常处理: 爬虫失败自动重试3次,数据校验失败进入"待修复队列"
- 权限控制: 管理员可查看全部数据,普通用户仅查看聚合统计结果
- 数据保留: 原始数据保留2年,聚合数据永久保留

## 七、技术架构建议

### 7.1 技术栈

- **后端**: Python (Django/Flask) + Java (Spring Boot)
- **数据库**: MySQL 8.0 + MongoDB (存储非结构化数据)
- **缓存**: Redis
- **消息队列**: RabbitMQ / Kafka
- **数据采集**: Scrapy + Selenium + API逆向
- **数据分析**: Pandas + NumPy + Scikit-learn
- **NLP**: jieba分词 + SnowNLP情感分析
- **可视化**: ECharts + D3.js
- **前端**: React + Ant Design

### 7.2 数据处理流程

```
数据采集层 → 数据清洗层 → 数据存储层 → 数据分析层 → 数据展示层
    ↓            ↓            ↓            ↓            ↓
  爬虫集群    ETL处理     MySQL/MongoDB   Spark/Pandas   Web前端
```
