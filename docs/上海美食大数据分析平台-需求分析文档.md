## 二、系统概述

### 2.1 系统定位

上海美食大数据分析平台是一个集数据采集、数据分析、数据可视化、智能推荐于一体的综合性平台,旨在为餐饮监管部门、餐饮从业者、市场研究机构和美食爱好者提供数据驱动的决策支持。

## 三、功能需求详细说明

### 3.1 数据采集模块

#### 3.1.1 多源数据爬取 (REQ-FUN-005-01)

**需求描述**: 系统应支持从多个数据源自动采集餐饮相关数据

**数据源列表**:
1. 大众点评 (dianping.com)
2. 美团 (meituan.com)
3. 小红书 (xiaohongshu.com)
4. 抖音/快手 (短视频平台)
5. 饿了么/美团外卖
6. 政府公开数据
7. 企业信用信息公示系统

**功能要求**:
- 支持配置化的爬虫任务管理
- 支持定时调度和手动触发
- 支持增量采集和全量采集
- 支持多种采集方式: Scrapy、Selenium、API逆向
- 支持代理IP池管理
- 支持User-Agent轮换
- 支持验证码识别(OCR)
- 支持反爬虫策略应对

**输入**: 爬虫任务配置(目标URL、采集规则、调度策略)
**输出**: 原始数据(JSON格式)
**优先级**: P0

#### 3.1.2 数据清洗 (REQ-FUN-005-02)

**需求描述**: 系统应对采集的原始数据进行清洗和标准化处理

**功能要求**:
- 数据去重: 基于餐厅名称+地址+电话的组合去重
- 数据格式化: 统一日期格式、价格格式、评分格式
- 异常值处理: 识别并处理异常数据(如价格为0、评分超出范围)
- 缺失值处理: 对缺失字段进行标记或填充
- 数据脱敏: 对个人敏感信息进行脱敏处理
- 数据验证: 验证数据完整性和合法性

**输入**: 原始数据
**输出**: 清洗后的结构化数据
**优先级**: P0

#### 3.1.3 爬虫监控 (REQ-FUN-005-03)

**需求描述**: 系统应提供爬虫运行状态监控和告警功能

**功能要求**:
- 实时监控爬虫运行状态
- 记录采集成功率、失败率
- 记录数据采集量和采集速度
- 失败自动重试(最多3次)
- 异常告警(邮件/短信/钉钉)
- 爬虫日志记录和查询

**输入**: 爬虫运行日志
**输出**: 监控报表、告警通知
**优先级**: P1

### 3.2 数据分析模块

#### 3.2.1 区域分析 (REQ-FUN-003-01)

**需求描述**: 系统应提供基于地理位置的餐饮数据分析功能

**功能要求**:
- 商圈餐厅密度分析: 统计各商圈餐厅数量分布
- 区域菜系分布: 分析不同区域的菜系偏好
- 区域消费水平: 统计各区域人均消费分布
- 商圈热力图: 可视化展示餐厅分布密度
- 地铁沿线餐厅分析: 分析地铁站周边餐厅分布

**输入**: 餐厅地理位置数据、商圈数据
**输出**: 区域分析报表、热力图数据
**优先级**: P0

#### 3.2.2 价格分析 (REQ-FUN-003-02)

**需求描述**: 系统应提供餐饮价格相关的分析功能

**功能要求**:
- 价格区间分布: 统计不同价格区间的餐厅数量
- 菜系价格对比: 对比不同菜系的平均消费水平
- 价格趋势分析: 分析餐饮价格的时间变化趋势
- 性价比排行: 综合评分和价格,计算性价比指数
- 价格异常检测: 识别价格异常的餐厅

**输入**: 餐厅价格数据、评分数据
**输出**: 价格分析报表、性价比排行榜
**优先级**: P0

#### 3.2.3 口碑分析 (REQ-FUN-003-03)

**需求描述**: 系统应提供基于用户评价的口碑分析功能

**功能要求**:
- 评分分布统计: 统计不同评分区间的餐厅数量
- 评论情感分析: 使用NLP技术分析用户评论情感倾向
- 热门菜品挖掘: 从评论中提取高频提及的菜品
- 服务质量分析: 分析服务相关评论的情感倾向
- 评论关键词提取: 提取评论中的高频关键词
- 评论词云生成: 生成评论关键词词云图

**输入**: 用户评论数据、评分数据
**输出**: 口碑分析报表、情感分析结果、词云图
**优先级**: P0

#### 3.2.4 趋势分析 (REQ-FUN-003-04)

**需求描述**: 系统应提供基于时间序列的趋势分析功能

**功能要求**:
- 菜系流行趋势: 分析不同菜系的热度变化趋势
- 新店开业趋势: 统计新开餐厅的数量和分布
- 消费趋势预测: 基于历史数据预测未来消费趋势
- 热门商圈变化: 分析商圈热度的时间变化
- 季节性分析: 分析餐饮消费的季节性特征

**输入**: 历史数据、时间序列数据
**输出**: 趋势分析报表、预测结果
**优先级**: P1

#### 3.2.5 竞品分析 (REQ-FUN-003-05)

**需求描述**: 系统应提供餐厅竞品分析功能

**功能要求**:
- 同类餐厅对比: 对比同一商圈、同一菜系的餐厅
- 优劣势分析: 分析餐厅在口味、环境、服务等方面的优劣势
- 市场份额分析: 分析不同品牌/连锁店的市场占有率
- 竞争格局分析: 分析特定区域的竞争态势
- SWOT分析: 生成餐厅的SWOT分析报告

**输入**: 餐厅数据、竞品数据
**输出**: 竞品分析报告
**优先级**: P0

### 3.4 智能推荐模块

#### 3.4.1 个性化推荐 (REQ-FUN-004-01)

**需求描述**: 系统应基于用户历史行为提供个性化餐厅推荐

**功能要求**:
- 基于协同过滤的推荐算法
- 考虑用户浏览历史、收藏记录
- 考虑用户评分偏好
- 考虑用户地理位置
- 实时更新推荐结果

**输入**: 用户行为数据、餐厅数据
**输出**: 推荐餐厅列表
**优先级**: P1

#### 3.4.2 相似餐厅推荐 (REQ-FUN-004-02)

**需求描述**: 系统应推荐与当前餐厅相似的其他餐厅

**功能要求**:
- 基于内容相似度的推荐算法
- 考虑菜系、价格、评分、地理位置等因素
- 计算餐厅相似度得分
- 返回Top N相似餐厅

**输入**: 当前餐厅ID、餐厅特征数据
**输出**: 相似餐厅列表
**优先级**: P1

#### 3.4.3 场景推荐 (REQ-FUN-004-03)

**需求描述**: 系统应根据用餐场景推荐合适的餐厅

**功能要求**:
- 支持多种场景: 约会、聚餐、商务、家庭、独自、团建、生日、下午茶
- 基于标签匹配的推荐算法
- 考虑场景特征(如约会需要环境好、商务需要包厢)
- 支持场景组合筛选

**输入**: 用餐场景、用户偏好
**输出**: 推荐餐厅列表
**优先级**: P1

## 五、数据模型设计

### 5.1 核心数据表

#### 5.1.1 餐厅信息表 (restaurant_info)

```sql
CREATE TABLE restaurant_info (
    restaurant_id VARCHAR(50) PRIMARY KEY COMMENT '餐厅唯一标识',
    restaurant_name VARCHAR(200) NOT NULL COMMENT '餐厅名称',
    address VARCHAR(500) COMMENT '餐厅地址',
    phone VARCHAR(20) COMMENT '联系电话(已脱敏)',
    business_hours VARCHAR(100) COMMENT '营业时间',
    avg_price DECIMAL(10,2) COMMENT '人均消费(元)',
    cuisine_type VARCHAR(50) COMMENT '菜系类型',
    district VARCHAR(50) COMMENT '所属行政区',
    business_area VARCHAR(100) COMMENT '所属商圈',
    latitude DECIMAL(10,6) COMMENT '纬度',
    longitude DECIMAL(10,6) COMMENT '经度',
    subway_distance INT COMMENT '地铁距离(米)',
    data_source VARCHAR(50) COMMENT '数据来源',
    crawl_time DATETIME COMMENT '采集时间',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_district (district),
    INDEX idx_business_area (business_area),
    INDEX idx_cuisine_type (cuisine_type),
    INDEX idx_location (latitude, longitude)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='餐厅基础信息表';
```

#### 5.1.2 评分评论表 (restaurant_review)

```sql
CREATE TABLE restaurant_review (
    review_id VARCHAR(50) PRIMARY KEY COMMENT '评论唯一标识',
    restaurant_id VARCHAR(50) NOT NULL COMMENT '餐厅ID',
    user_id VARCHAR(50) COMMENT '用户ID(已脱敏)',
    rating_overall DECIMAL(3,1) COMMENT '综合评分',
    rating_taste DECIMAL(3,1) COMMENT '口味评分',
    rating_environment DECIMAL(3,1) COMMENT '环境评分',
    rating_service DECIMAL(3,1) COMMENT '服务评分',
    review_content TEXT COMMENT '评论内容',
    review_time DATETIME COMMENT '评论时间',
    like_count INT DEFAULT 0 COMMENT '点赞数',
    sentiment_score DECIMAL(3,2) COMMENT '情感得分',
    data_source VARCHAR(50) COMMENT '数据来源',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_restaurant_id (restaurant_id),
    INDEX idx_review_time (review_time),
    FOREIGN KEY (restaurant_id) REFERENCES restaurant_info(restaurant_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='评分评论表';
```

## 六、接口需求

### 6.1 RESTful API设计原则

- 使用HTTP方法: GET(查询)、POST(创建)、PUT(更新)、DELETE(删除)
- 使用标准HTTP状态码: 200(成功)、400(请求错误)、401(未授权)、404(未找到)、500(服务器错误)
- 统一响应格式: `{"code": 200, "message": "success", "data": {...}}`
- 支持分页: `?page=1&size=20`
- 支持排序: `?sort=rating&order=desc`
- 支持筛选: `?cuisine=本帮菜&district=徐汇区`

### 6.2 核心API接口

#### 6.2.1 餐厅搜索接口

```
GET /api/v1/restaurants/search
```

**请求参数**:
- keyword: 关键词(可选)
- district: 行政区(可选)
- business_area: 商圈(可选)
- cuisine_type: 菜系(可选)
- price_min: 最低价格(可选)
- price_max: 最高价格(可选)
- rating_min: 最低评分(可选)
- page: 页码(默认1)
- size: 每页数量(默认20)
- sort: 排序字段(默认rating)
- order: 排序方式(asc/desc,默认desc)

**响应示例**:
