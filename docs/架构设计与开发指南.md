# 上海美食大数据分析平台 — 架构设计与开发指南

> 本文档面向所有参与开发的同学，用通俗的语言把项目"是什么、怎么转、怎么写"讲清楚。
> 建议先通读一遍再动手，遇到不确定的地方回来查。

---

## 一、一句话理解这个项目

我们要做的事情可以概括为一条流水线：

```
爬数据 → 洗数据 → 存数据 → 分析数据 → 展示数据
```

用户打开网页，能搜餐厅、看分析图表、下载报告、收到推荐；
管理员在后台能管爬虫、管用户、看数据质量。

## 二、整体架构：四个独立模块 + 两个基础设施

把项目想象成一家工厂，有四个车间和两个仓库：

```
┌──────────────────────────────────────────────────────────────┐
│                        用户的浏览器                            │
└──────────────┬───────────────────────────────────────────────┘
               │ HTTP 请求 (所有接口以 /api/v1 开头)
               ▼
┌──────────────────────────┐      ┌──────────────────────────┐
│   frontend/ (前端车间)    │      │   crawler/ (采集车间)     │
│   React + Ant Design     │      │   Scrapy + Selenium       │
│   负责：页面、图表、交互   │      │   负责：爬取、清洗、入库   │
└──────────────┬───────────┘      └────────────┬─────────────┘
               │                                │
               ▼                                ▼
┌──────────────────────────────────────────────────────────────┐
│                   backend/ (后端车间)                          │
│                   FastAPI + Celery                            │
│   负责：接口、认证、调度分析任务、生成报告                       │
│                                                              │
│   内部调用 ──→  analysis/ (分析车间)                           │
│                 Pandas + NLP + 推荐算法                       │
│                 负责：统计、情感分析、推荐计算                   │
└──────────┬──────────────────────────┬────────────────────────┘
           │                          │
           ▼                          ▼
    ┌─────────────┐           ┌─────────────┐
    │  MySQL 8.0  │           │    Redis     │
    │  (数据仓库)  │           │  (缓存仓库)  │
    └─────────────┘           └─────────────┘
```

### 各车间职责速查

| 模块 | 目录 | 一句话职责 | 主要技术 |
|------|------|-----------|---------|
| 前端 | `frontend/` | 用户看到的所有页面和交互 | React 19, Ant Design, ECharts, 高德地图 |
| 后端 | `backend/` | 接收前端请求，返回数据，调度任务 | FastAPI, SQLAlchemy, Celery |
| 采集 | `crawler/` | 从大众点评等网站爬数据，清洗后存入数据库 | Scrapy, Selenium |
| 分析 | `analysis/` | 纯计算模块，被后端调用，不直接对外 | Pandas, Scikit-learn, jieba, SnowNLP |

### 两个仓库

| 基础设施 | 作用 |
|---------|------|
| MySQL | 存所有结构化数据：餐厅、评论、菜品、用户、任务 |
| Redis | 两个用途：① 缓存分析结果（避免重复计算）② 作为 Celery 的消息队列 |

## 三、数据怎么流动的？

理解数据流是理解整个项目的关键。下面用三个典型场景来说明。

### 场景 1：爬虫采集数据（每天凌晨 2 点自动执行）

```
大众点评网页
    │
    ▼
Spider（爬虫）抓取 HTML/JSON，提取字段
    │
    ▼
DataCleanPipeline（清洗）
    - 去重（同一家店不重复存）
    - 格式统一（日期、价格、评分）
    - 脱敏（手机号变成 138****1234）
    │
    ▼
MySQLPipeline（入库）
    - 写入 restaurant_info / restaurant_review / dish_info 表
```

### 场景 2：用户搜索餐厅

```
用户在搜索页输入"徐家汇 本帮菜"，点搜索
    │
    ▼
前端 SearchBar 组件 → 调用 api/restaurants.js 的 searchRestaurants()
    │
    ▼
Axios 发送 GET /api/v1/restaurants/search?keyword=本帮菜&business_area=徐家汇
    │
    ▼
后端 restaurants.py 路由接收请求
    → 调用 restaurant_service.search_restaurants()
    → SQLAlchemy 构建动态查询条件，查 MySQL
    → 返回 JSON：{ code: 200, data: { total: 35, items: [...] } }
    │
    ▼
前端拿到数据，渲染餐厅卡片列表
```

### 场景 3：生成市场分析报告（耗时操作，异步处理）

```
用户选择"浦东新区 + 日料 + 近3个月"，点击生成报告
    │
    ▼
前端 POST /api/v1/reports/market  →  后端 reports.py 路由
    │
    ▼
report_service.create_market_report_task()
    - 在 report_task 表写入一条记录（状态：pending）
    - 提交一个 Celery 异步任务
    - 立即返回 task_id 给前端（用户不用等）
    │
    ▼
Celery Worker 在后台执行 generate_market_report()：
    1. 从 MySQL 查询符合条件的数据
    2. 调用 analysis/ 模块做统计分析
    3. 用 matplotlib 生成图表
    4. 用 reportlab 生成 PDF（或 python-docx 生成 Word）
    5. 更新任务状态为 completed，记录文件路径
    │
    ▼
前端轮询 GET /api/v1/reports/tasks/{task_id} 查状态
    → 状态变成 completed 后，显示下载按钮
    → 用户点击下载 PDF
```

**为什么要用 Celery？** 因为生成报告可能要几分钟，如果在接口里同步执行，用户会一直等着，浏览器可能超时。Celery 把耗时任务丢到后台队列，接口立刻返回，用户体验好得多。

## 四、后端架构详解

后端是整个系统的"大脑"，采用经典的分层架构：

```
请求进来
    │
    ▼
┌─ api/v1/ ─────────────────────┐  ← 路由层：接收请求，校验参数
│  restaurants.py  analysis.py  │
│  reports.py  recommend.py     │
│  auth.py     admin.py         │
└───────────────┬───────────────┘
                │ 调用
                ▼
┌─ services/ ───────────────────┐  ← 服务层：业务逻辑
│  restaurant_service.py        │
│  analysis_service.py          │
│  report_service.py            │
│  recommend_service.py         │
│  auth_service.py              │
└───────────────┬───────────────┘
                │ 调用
                ▼
┌─ models/ ─────────────────────┐  ← 模型层：数据库表映射
│  restaurant.py  review.py     │
│  dish.py  user.py             │
│  crawl_task.py                │
└───────────────────────────────┘
```

### 各层职责

- **路由层**（`api/v1/`）：只做三件事 —— 接收参数、调用服务、返回结果。不写业务逻辑。
- **服务层**（`services/`）：所有业务逻辑都在这里。比如"搜索餐厅时怎么拼查询条件""生成报告时调用哪些分析函数"。
- **模型层**（`models/`）：用 SQLAlchemy ORM 定义数据库表结构。一个 `.py` 文件对应一张表。
- **Schema 层**（`schemas/`）：用 Pydantic 定义请求和响应的数据格式。FastAPI 会自动做参数校验。
- **Core 层**（`core/`）：横切关注点 —— 安全（JWT）、依赖注入（数据库会话）、异常处理。

### 认证流程

```
用户登录 POST /api/v1/auth/login
    │
    ▼
auth_service.authenticate_user()
    - 查数据库找用户
    - BCrypt 验证密码
    - 生成 JWT Token（包含 user_id 和 role，有效期 24 小时）
    │
    ▼
返回 { access_token: "eyJ...", token_type: "bearer" }
    │
    ▼
前端把 token 存到 localStorage
之后每次请求，Axios 拦截器自动加上 Header：
    Authorization: Bearer eyJ...
    │
    ▼
后端 core/deps.py 的 get_current_user() 解析 Token，得到当前用户
需要管理员权限的接口，再用 require_role("admin") 做二次校验
```

### 统一响应格式

所有接口返回同一种 JSON 结构，前端处理起来很方便：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

错误时：

```json
{
  "code": 401,
  "message": "Token已过期，请重新登录",
  "data": null
}
```

## 五、前端架构详解

前端是一个标准的 React 单页应用（SPA），用 Vite 构建。

### 目录结构逻辑

```
src/
├── views/          页面组件（一个路由对应一个文件）
├── components/     可复用组件（图表、地图、搜索栏等）
├── api/            后端接口调用（一个后端模块对应一个文件）
├── store/          全局状态管理（Zustand）
├── router/         路由配置
└── utils/          工具函数（Axios 封装等）
```

### 前端和后端怎么通信？

```
组件（如 Search.jsx）
    │ 调用
    ▼
api 层（如 api/restaurants.js 的 searchRestaurants）
    │ 调用
    ▼
utils/request.js（Axios 实例）
    │ 发送 HTTP 请求
    ▼
Vite 开发服务器代理（/api → localhost:8000）
    │
    ▼
后端 FastAPI
```

**开发时**，前端跑在 3000 端口，后端跑在 8000 端口。`vite.config.js` 里配了代理，前端请求 `/api/v1/xxx` 会自动转发到后端，不会有跨域问题。

### 状态管理（Zustand）

我们用 Zustand 而不是 Redux，因为它更轻量，写法更简单。项目里有三个 Store：

| Store | 管什么 | 哪些页面用 |
|-------|--------|-----------|
| `useUserStore` | 登录状态、Token、用户信息、角色 | 几乎所有页面（导航栏显示用户名、权限判断） |
| `useSearchStore` | 搜索条件、搜索结果、分页 | Search 页面 |
| `useAnalysisStore` | 各分析模块的缓存数据 | Analysis 页面 |

### 图表怎么画？

所有数据可视化都用 ECharts，通过 `echarts-for-react` 这个 React 封装库来用。

我们把常用图表封装成了独立组件，放在 `components/charts/` 下：

| 组件 | 用途 | 用在哪 |
|------|------|--------|
| `HeatMap.jsx` | 餐厅分布热力图（叠加在高德地图上） | 区域分析 |
| `TrendChart.jsx` | 折线图/面积图（时间趋势） | 趋势分析 |
| `PriceChart.jsx` | 柱状图/饼图/折线图（价格相关） | 价格分析 |
| `RadarChart.jsx` | 雷达图（多维度对比） | 餐厅详情、竞品分析 |
| `WordCloud.jsx` | 词云图（关键词频率） | 口碑分析 |

使用方式很简单：父组件从后端拿到数据，通过 props 传给图表组件，图表组件负责渲染。

## 六、爬虫模块详解

爬虫是整个平台的数据源头。我们用 Scrapy 框架，它的工作流程是固定的：

```
Spider（定义怎么爬）
    │ 产出 Item
    ▼
Middleware（中间件，处理请求/响应）
    - 换 User-Agent（假装是不同浏览器）
    - 换代理 IP（防止被封）
    - 处理重试
    │
    ▼
Pipeline（管道，处理数据）
    - DataCleanPipeline：清洗
    - MySQLPipeline：入库
    - DataQualityPipeline：质量监控
```

### 四个爬虫各负责什么

| 爬虫 | 数据源 | 采集内容 | 更新频率 |
|------|--------|---------|---------|
| `dianping_spider` | 大众点评 | 餐厅信息、评分、评论、菜品、标签 | 每日增量 + 每周全量 |
| `meituan_spider` | 美团 | 餐厅信息、销量、优惠、评价、配送 | 每日增量 |
| `xiaohongshu_spider` | 小红书 | 探店笔记、图片、标签、互动 | 每日热门 |
| `gov_data_spider` | 政府网站 | 食品许可证、抽检结果、行政处罚 | 每月 |

### 反爬虫怎么应对

大众点评的反爬最复杂，主要有三道防线：

1. **字体加密**：页面上的数字（评分、价格）用自定义字体渲染，直接爬到的是乱码。我们需要下载字体文件，解析字符映射关系。
2. **CSS 偏移**：部分数字通过 CSS 位移拼接，需要计算实际值。
3. **验证码**：频繁访问会触发滑块验证码，用 ddddocr 库做 OCR 识别。

这些逻辑都在 `crawler/shanghai_food/utils/anti_crawl.py` 里。

### 重要约束

- 采集频率 ≤ 1次/秒（`DOWNLOAD_DELAY = 1.0`），这是法律合规要求
- 遵守 robots.txt（`ROBOTSTXT_OBEY = True`）
- 个人信息必须脱敏后才能入库

## 七、分析模块详解

`analysis/` 是纯 Python 计算模块，不直接对外提供接口，而是被后端的 `services/analysis_service.py` 调用。

### 模块结构

```
analysis/
├── area_analysis.py          区域分析（商圈密度、菜系分布、热力图数据）
├── price_analysis.py         价格分析（区间分布、趋势、性价比排行）
├── review_analysis.py        口碑分析（评分分布、情感分析、词云）
├── trend_analysis.py         趋势分析（菜系趋势、新店趋势、预测）
├── competitor_analysis.py    竞品分析（对比、SWOT）
├── nlp/                      自然语言处理
│   ├── sentiment.py          情感分析（SnowNLP）
│   ├── keyword_extract.py    关键词提取（jieba TF-IDF）
│   └── dish_mining.py        从评论中挖掘菜品名
└── recommend/                推荐算法
    ├── collaborative_filter.py  协同过滤（"和你口味相似的人还喜欢..."）
    ├── content_based.py         内容推荐（"和这家店类似的还有..."）
    └── scene_recommend.py       场景推荐（"约会适合去..."）
```

### NLP 怎么工作的？

以情感分析为例，用户评论"味道不错，环境舒适，就是服务有点慢"会经过这样的处理：

```
原始评论
    │
    ▼
jieba 分词 → ["味道", "不错", "环境", "舒适", "服务", "有点", "慢"]
    │
    ▼
SnowNLP 整体情感打分 → 0.78（偏正面）
    │
    ▼
方面级情感分析（基于 config/dictionaries/sentiment_dict.py 关键词匹配）：
    - 味道：正面（命中"不错"）
    - 环境：正面（命中"舒适"）
    - 服务：负面（命中"慢"）
```

### 推荐算法简单解释

**协同过滤**：找到和你评分习惯相似的用户群体，把他们喜欢但你还没去过的餐厅推荐给你。核心是构建"用户×餐厅"评分矩阵，计算用户间的余弦相似度。

**内容推荐**：把每家餐厅变成一个特征向量（菜系、价格、评分、区域、标签），计算餐厅间的相似度。你正在看川菜馆 A，系统找出和 A 最像的 B、C、D 推荐给你。

**场景推荐**：根据"约会""聚餐""商务"等场景标签匹配餐厅。标签来自大众点评的原始数据和评论关键词挖掘。

## 八、数据库设计

### 核心表关系

```
restaurant_info（餐厅）  1 ──→ N  restaurant_review（评论）
                         1 ──→ N  dish_info（菜品）

sys_user（用户）         N ←──→ N  restaurant_info（收藏关系，通过 user_favorites 中间表）

crawl_task（爬虫任务）   1 ──→ N  crawl_log（运行日志）
```

### 关键字段说明

**restaurant_info** 表里有几个容易混淆的字段：
- `district`：行政区（如"徐汇区"），上海共 16 个
- `business_area`：商圈（如"徐家汇"），一个行政区有多个商圈
- `data_source`：数据来自哪个平台（dianping / meituan / xiaohongshu / gov）
- `crawl_time`：这条数据是什么时候爬到的

**restaurant_review** 表里的 `sentiment_score`：
- 范围 -1 到 1，由 NLP 模块计算后回填
- > 0.6 算正面，< -0.4 算负面，中间算中性

### 业务计算字段（不存数据库，实时计算）

| 字段 | 计算公式 | 用途 |
|------|---------|------|
| 价格等级 | ≤50 经济型 / 51-100 中档 / 101-200 中高档 / >200 高档 | 筛选、统计 |
| 热门度 | 评分×0.3 + 评论数×0.3 + 月销量×0.4 | 热门排序 |
| 性价比指数 | (评分/5) ÷ (人均/100) | 性价比排行 |

## 九、配置与字典

### 环境配置

项目有两个配置文件需要在本地创建（不提交到 Git）：

1. `.env`（从 `.env.example` 复制）—— 数据库密码、API Key 等敏感信息
2. `config/config.yaml`（从 `config/config.example.yaml` 复制）—— 应用级配置

### 业务字典

`config/dictionaries/` 下有四个字典文件，是分析和爬虫模块的共享知识库：

| 文件 | 内容 | 谁在用 |
|------|------|--------|
| `cuisine_dict.py` | 菜系分类 + 同义词 + 热门菜品 | 爬虫分类、搜索匹配、菜品挖掘 |
| `area_dict.py` | 16 个行政区 → 商圈映射 | 区域分析、地理归类 |
| `scene_dict.py` | 用餐场景 + 关键词 + 特色服务 | 场景推荐、标签匹配 |
| `sentiment_dict.py` | 正面/负面关键词（按口味/环境/服务/性价比分类） | 方面级情感分析 |

如果需要新增菜系、商圈或关键词，直接改这些字典文件即可，不需要改代码逻辑。

## 十、开发环境搭建

### 方式一：Docker 一键启动（推荐）

```bash
cp .env.example .env          # 编辑填入实际配置
cp config/config.example.yaml config/config.yaml
docker compose up -d --build   # 启动所有服务
```

启动后：
- 前端：http://localhost:3000
- 后端 API：http://localhost:8000
- 后端文档：http://localhost:8000/docs（FastAPI 自动生成的 Swagger UI）

### 方式二：手动分别启动

**1. 启动 MySQL 和 Redis**（自行安装或用 Docker 单独启动）

**2. 初始化数据库**
```bash
mysql -u root -p < scripts/init_db.sql
```

**3. 启动后端**
```bash
cd backend
pip install -r requirements.txt
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

**4. 启动 Celery Worker**（新开一个终端）
```bash
cd backend
celery -A app.main worker --loglevel=info
```

**5. 启动前端**
```bash
cd frontend
npm install
npm run dev
```

**6. 运行爬虫**（按需）
```bash
cd crawler
pip install -r requirements.txt
scrapy crawl dianping_spider
```

## 十一、开发分工建议

项目天然适合按模块分工，各模块之间通过明确的接口交互：

| 角色 | 负责模块 | 需要掌握的技术 | 和谁对接 |
|------|---------|---------------|---------|
| 前端开发 | `frontend/` | React, Ant Design, ECharts | 和后端对接 API 格式 |
| 后端开发 | `backend/` | FastAPI, SQLAlchemy, Celery | 和前端约定接口，调用分析模块 |
| 爬虫开发 | `crawler/` | Scrapy, Selenium, 反爬技术 | 产出数据写入 MySQL，和后端共享数据库 |
| 算法开发 | `analysis/` | Pandas, Scikit-learn, NLP | 提供函数给后端 service 层调用 |

### 协作约定

1. **接口格式**：所有 API 遵循 RESTful 风格，统一响应格式 `{code, message, data}`
2. **数据库变更**：修改表结构前先在群里沟通，更新 `models/` 和 `scripts/init_db.sql`
3. **字典更新**：新增菜系/商圈/关键词，改 `config/dictionaries/` 下对应文件
4. **分支策略**：功能分支从 master 拉出，命名 `feature/模块名-功能描述`，完成后提 PR

## 十二、常见问题

**Q：前端请求后端报 CORS 错误？**
开发模式下不应该出现，因为 Vite 配了代理。如果出现，检查请求是否写了完整 URL（如 `http://localhost:8000/api/...`），应该只写路径 `/api/v1/...`。

**Q：爬虫被目标网站封了怎么办？**
1. 检查 `DOWNLOAD_DELAY` 是否 ≥ 1.0
2. 检查代理 IP 池是否正常
3. 查看 `anti_crawl.py` 里的验证码处理是否生效
4. 降低 `CONCURRENT_REQUESTS`

**Q：分析模块怎么单独测试？**
`analysis/` 是纯 Python 模块，可以直接 import 调用。只要本地能连 MySQL 读到数据就行，不依赖后端服务。

**Q：报告生成任务一直是 pending 状态？**
检查 Celery Worker 是否启动，以及 Redis 是否正常连接（Celery 用 Redis 做消息队列）。
